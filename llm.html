Big Picture

AI → ML → Deep Learning → GenAI → LLM

LLMs are not a separate universe. They sit inside the broader AI family tree.

AI

Machines performing tasks that seem intelligent: chess engines, route planners, fraud detection, conversational assistants.

Machine Learning

Systems learn patterns from examples rather than hand-coded rules. Give data, not instructions.

Deep Learning

Neural networks with many layers learn progressively complex representations of the input.

Generative AI

AI that creates new text, code, images, audio, or structured outputs — not just classifying.

LLM

A large neural language model trained on massive text/code to predict and generate useful language.

Agent

LLM + tools + memory + workflow control + approvals. The Rudhra platform you're building on.

Key contrast: Traditional software says "give me exact rules." ML says "give me examples." LLMs say "give me language context — I'll generate the next useful token based on learned patterns across the internet."
Recent History of Language AI

From word counts to frontier models

Language AI has always had one hard problem: language is messy, contextual, and ambiguous. This history is a story of better representations — from counting words, to dense vectors, to attention, to instruction-following giants.

~2000
Bag-of-Words & TF-IDFText as counts; simple, fast, but loses word order and deep semantics entirely.
2013
Word2VecWords become dense vectors. Semantically similar words appear close in mathematical space. "King − Man + Woman ≈ Queen"
2017
"Attention Is All You Need" — TransformerGoogle's paper. Attention replaces recurrence as the core mechanism. Modern LLMs trace directly here.
2018–20
BERT, GPT, GPT-2, GPT-3, T5Encoder, decoder, and encoder-decoder models scale rapidly. GPT-3 (175B params) shocks the research community.
2022
ChatGPT (GPT-3.5 + RLHF)Instruction-following chat interface makes LLMs mainstream. 100M users in 2 months.
2023–24
GPT-4, Claude 3.x, Gemini 1.x, Llama-2/3Reasoning, coding, multimodal, and long-context (1M+ tokens) improve dramatically.
2025+
GPT-5, Gemini 2.5/3, Claude 4.x, o-seriesFrontier models push harder on agentic reasoning, native tool use, and code generation. Context windows exceed 1M tokens.
Key insight: The big jump wasn't only bigger models. It was better representations, better architectures for handling context, and RLHF to teach models to follow human instructions.
From Text to Tokens

LLMs do not read words — they read tokens

Text is split into tokens before the model sees it. A token can be a whole word, a word fragment, a punctuation mark, a whitespace, a newline, or a code symbol. The same sentence takes different token counts depending on the language, tokenizer, and vocabulary.

Meaning as Mathematics

Embeddings: turning tokens into vectors of meaning

An embedding maps a token, sentence, or document to a list of numbers. Semantically similar things cluster together in that high-dimensional space. This is why semantic search works — "NullPointerException in OrderService" and "null reference error in order module" are different strings but close vectors.

Token embedding

Learned per-token vector inside the model. Updated during training to capture contextual meaning.

Sentence embedding

A single vector for a full sentence — useful for semantic search and classification.

Document embedding

Chunk-level vector for RAG. Lets you search Confluence, Jira, or code by meaning, not keywords.

Engineering insight: Semantic search in your RAG pipeline uses embeddings under the hood. Two different wordings of the same bug report → nearby vectors → same search result.
Neural Networks and Vector Math

The essence: transform vectors through learned layers

Vector

A list of numbers encoding meaning. cat → [0.12, -0.45, 0.88, …] — typically 768 to 4096 dimensions.

Neural network

A large adjustable machine that transforms input vectors into progressively more useful output vectors.

Matrix multiplication

The engine of every layer: input vector × weight matrix = transformed vector. Billions of these happen per token.

Training

Predict → measure error → nudge weights slightly. Repeat on trillions of tokens until the model captures language.

Vector additioncombine meanings
Dot productmeasure similarity
Matrix multiplytransform context
Softmaxscores → probabilities
Backproplearn from error
Mental model: Don't teach the calculus. The key insight is: language → vectors → neural layers transform those vectors → the model learns which transformations produce good predictions.
How Learning Happens

Training: predict → measure error → adjust weights

During pre-training the model sees the next token masked and tries to predict it. Wrong prediction → error signal → tiny weight adjustment. This repeats across trillions of tokens, encoding grammar, facts, code patterns, and reasoning heuristics.

Input: The capital of France is ___
Model predicts: London
Training target: Paris
Loss: prediction was wrong → adjust
Repeat: billions of examples

Three training phases

Pre-training
Massive unlabelled text/code. Learns broad language + world knowledge.
Supervised Fine-tuning (SFT)
Human-written (prompt, ideal response) pairs. Teaches instruction-following.
RLHF / Preference tuning
Human rankers prefer response A over B. Reinforcement nudges model toward preferred outputs.
Presenter: "Pre-training builds the knowledge. SFT builds the assistant behaviour. RLHF polishes style and safety."
Before Transformers

Encoder–Decoder and the bottleneck problem

Input: I love llamas
Encoder RNN: process left-to-right
Context vector: one fixed-size vector
Decoder RNN: generate output
Output: Ik hou van lama's

The bottleneck problem

For short sentences this works. For a long Spring Boot stack trace, a full JIRA ticket, or a multi-file PR — compressing everything into one fixed vector loses details mid-sequence.


The conceptual bridge

Attention was originally added to RNN seq2seq to let the decoder look back at all encoder states — not just the final compressed one. This idea grew into the full Transformer.

The key failure mode was called the vanishing gradient — information about early tokens faded as sequences got longer. Attention solved this by allowing direct connections between any two positions.
The Transformer Breakthrough

Attention: every token can look at every other token

The Transformer computes an attention score between every pair of tokens in the sequence. This lets the model directly resolve references ("it" → "the API"), handle long dependencies, and weigh what's relevant for each position — all in parallel.

Darker cell = higher attention weight. The model attends to the tokens most relevant for generating or contextualising the focused word.

Without attention

The model processes left-to-right and may lose early context by the time it reaches distant tokens.

With attention (multi-head)

"it" directly attends to "API". "500" attends to "returning". Different heads capture different relationship types simultaneously.

Inside a Transformer LLM

Tokenizer → Embeddings → Blocks → LM Head

Every token generation requires one full forward pass through the entire model stack. The LM head converts the final layer's output into a probability distribution over the entire vocabulary (~32K–100K+ tokens).

One Transformer block contains

Multi-Head Self-Attention — compute relevance between all token pairs
Feed-Forward Network — per-token non-linear transformation
Layer Norm + Residual — training stability, preserves gradient flow

Scale in numbers

96Transformer blocks (GPT-4 class)
~1TParameters (frontier models)
1M+Token context window (Claude 3.5+)

Instruction Interface

System prompts and the chat turn structure

Instruction-tuned LLMs expect a structured conversation format. The system prompt sets the model's role, persona, and constraints. Then alternating user/assistant turns form the context window the model reasons over.

🔧 System You are a senior Java engineer reviewing pull requests. You are concise and direct. Always flag: (1) security issues first, (2) performance anti-patterns, (3) code style. Never hallucinate — if unsure, say so explicitly.
👤 User Review this method:

@GetMapping("/orders/{id}")
public Order getOrder(@PathVariable String id) {
  return db.query("SELECT * FROM orders WHERE id = " + id);
}
🤖 Assistant 🚨 Critical security issue: SQL injection vulnerability. User-controlled `id` is concatenated directly into the query. Use parameterised queries: db.query("SELECT * FROM orders WHERE id = ?", id).

Also consider: validate `id` format, add authorisation check (can this user see this order?), and handle the null case.

Why the system prompt matters

It's the only persistent context across all turns. It shapes persona, output format, domain focus, and safety guardrails. In agent systems (like Rudhra), the system prompt carries tool definitions and workspace context.

How instruction tuning enables this

SFT training teaches the model to follow the structure. RLHF teaches it to be useful and safe within that structure. Without these training phases, the model would just continue text — not behave like an assistant.

Agent note (Rudhra-specific): In agentic workflows the system prompt also carries tool schemas, workspace ID, available connectors, and HITL checkpoint instructions. Getting the system prompt right is the primary tuning lever for agent behaviour.
Retrieval-Augmented Generation

RAG: giving the model current, private knowledge

LLMs have a training cutoff and cannot see your internal docs, latest Confluence pages, or production configs. RAG solves this: retrieve relevant chunks from your knowledge base and inject them into the context before generation.

1
User query arrives"How does the OrderService HITL checkpoint work in our architecture?"
2
Embed the queryQuery → embedding vector via the same model (e.g. text-embedding-3-large). Same space as the doc chunks.
3
Vector search over knowledge baseFind top-K most similar chunks in Supabase pgvector / Pinecone / Weaviate. "HITL checkpoint", "OrderService architecture", "Rudhra checkpoint store" → cosine similarity match.
4
Inject retrieved context into promptPrepend the retrieved Confluence pages / code snippets to the system prompt or user message. Now the model has real, current information.
5
LLM generates grounded answerResponse is grounded in your actual docs. Hallucination risk drops dramatically. Model can also cite the source chunk.
When to use RAG: Private/internal knowledge, frequently changing content (docs, tickets, configs), large corpora that don't fit in context, and when you need source citations.
Extending LLM Capabilities

Tool use: LLMs calling your APIs

Function calling allows the model to emit a structured JSON call instead of free text when it determines it needs real-time information or must take an action. Your code executes the function and returns the result. The model incorporates it into its next response.

Tool definition (JSON Schema)

{ "name": "get_jira_issue", "description": "Fetch a Jira ticket by ID", "parameters": { "type": "object", "properties": { "issue_key": { "type": "string", "description": "e.g. RUD-120" } }, "required": ["issue_key"] } }

Round-trip flow

LLM decidesI need the current status of RUD-120 to answer this.
↓
Function call emittedget_jira_issue({ "issue_key": "RUD-120" })
↓
Your code executes + returns{ "status": "Done", "assignee": "Natarajan", "points": 5 }
↓
LLM incorporates resultRUD-120 is Done (5 points), assigned to Natarajan. The v0.10 milestone review is complete.
Security note for agent builders: Tool inputs come from the LLM — treat them as untrusted user input. Validate, sanitise, and apply the principle of least privilege. Prompt injection can hijack tool calls.
Probability and Sampling

How the model picks the next token

The LM head produces scores for every token in the vocabulary (~32K+). Sampling settings determine whether the model always picks the most likely token (greedy) or samples from the distribution with some randomness (temperature > 0).

Temperature

Low (0.1)

Engineering rule: Set temperature low (0–0.2) for code generation, structured JSON output, and fact extraction. Higher for creative naming, documentation drafts, and idea generation.
Capacity & Limits

Context window vs model parameters

Context

What the model can see now

Prompt, system prompt, tool definitions, RAG chunks, chat history, generated output — all consume context tokens. When the window fills, older tokens are dropped.

Params

What the model learned in training

7B, 70B, 400B+ adjustable weights. More parameters → richer language understanding, stronger reasoning, better code generation — but higher inference cost.

Large context helps when…

Including long stack traces, full Jira epics, code files, meeting transcripts, or long conversation history. Claude 3.5 supports 200K tokens (~150K words).

More parameters help when…

The task requires stronger multi-step reasoning, complex code generation, nuanced language understanding, or reliable factual recall.

Context gives evidence. Parameters give capability. Validation gives trust.

Reality Check

Why LLMs can be useful and still wrong

LLMs generate likely text based on learned patterns and current context. They do not have access to your production state, real-time data, private architecture, or post-training events unless you explicitly provide them.

Hallucination

The model produces confident but false content — invented APIs, wrong method signatures, fabricated references, or plausible-sounding but incorrect reasoning.

Missing context

Without logs, code, configs, and error messages, answers are generic speculation. Garbage in → garbage out applies here harder than anywhere.

Stale knowledge

Training cutoff means the model may not know the latest Spring Boot version, Angular 19 APIs, or your new microservice — unless you include that information.

Prompt injection

Malicious content in RAG chunks or tool responses can hijack model behaviour. Critical for agent builders: never trust LLM tool inputs without validation.

Bad prompt

"Why did production fail?"

Better prompt

"Here is the stack trace [paste], the deployment diff [paste], and the method signature [paste]. Identify likely causes and rank by confidence."

Team rule: AI output is a high-quality draft, not a final decision. Important outputs need verification through tests, source documents, peer review, or logs.
Engineer's Playbook

Prompting patterns that actually work

Good prompts are not magic words — they're information engineering. Give the model role, context, format constraints, and examples.

Role prompting

Set a persona

// System prompt You are a senior Java engineer specialising in Spring Boot microservices and performance. Be direct. Prioritise security issues first.

Narrows the model's prior — it writes at a higher technical level, uses correct terminology, and focuses on what matters to engineers.

Few-shot

Show examples

// Provide input-output pairs Input: getOrderById Output: "Retrieves order by ID" Input: processPaymentRefund Output: "Processes refund for payment" Input: validateUserPermissions Output: ___

Two or three examples dramatically improve output format and style consistency — especially for structured generation tasks.

Chain-of-thought

Ask for reasoning steps

// Add to the end of your prompt: "Think step by step. First identify what the error message tells us. Then consider the call stack. Then propose a fix."

Forces the model to reason before concluding. Significantly reduces hallucination on multi-step debugging and architecture problems.

Structured output

Specify exact format

// Tell the model exactly what to return "Return valid JSON only: { issue: string, severity: 'low'|'medium'|'critical', fix: string, confidence: number (0-1) }"

Essential for integration with your services. Combine with low temperature and parse defensively — never trust raw LLM JSON without schema validation.

Golden rule: The more context and constraints you give, the less the model has to guess. Every guess is a hallucination risk.
Session Close

What to carry forward

Text → tokens → vectors

The model never reads English — it reads token IDs mapped to vectors in a high-dimensional space.

Transformers use attention

Every token can attend to every other. This solves the long-range dependency problem and is why modern LLMs handle full codebases.

Generation is probabilistic

Each token is sampled from a distribution. Temperature controls creativity. Low = deterministic; high = imaginative.

Context is your primary lever

System prompts, RAG, tool results, and examples all shape context. Better context → better output. Validate everything.

For debugging: paste the actual error, stack trace, relevant code, and config. Ask for ranked hypotheses with reasoning.
For code generation: specify language, framework version, constraints, and show a before/after example if you can.
For agents (Rudhra): the system prompt carries the agent's identity and tools. HITL checkpoints + validation loops are your safety net.
Always: treat LLM output as a high-quality draft. Run tests, review diffs, verify facts against source documents.
