Step by Step

From raw text to a vector

Every embedding model runs the same conceptual pipeline. The architecture and dimension count differ, but the flow — text → tokens → IDs → lookup → transform → pool → vector — is universal.

1
Raw text arrives The model never reads raw English. It receives a string and immediately passes it to the tokeniser.
"NullPointerException in OrderService.processPayment()"
2
Tokeniser splits into subword tokens BPE or WordPiece splits the string. Rare or compound words become multiple pieces.
3
Tokens → integer IDs Each token maps to a row index in the model's vocabulary (typically 30K–100K entries).
[ 7670, 13924, 1999, 6150, 18095, 6124, 1006, 1007 ]
4
Embedding lookup — each ID → a dense vector The model looks up each token ID in a learned embedding matrix (vocab_size × dim). Each token becomes a row of floating-point numbers. For BERT-base: 768 numbers per token.

5
Transformer layers refine the representation The token vectors pass through N Transformer blocks. Each block uses attention so every token can incorporate context from every other token. "NullPointerException" now knows it appears in a Java method-call context.
6
Pooling → one sentence vector The per-token vectors are collapsed into a single sentence vector. Common strategies: take the [CLS] token output (BERT), mean-pool all tokens, or use a special pooling head.

Cosine Similarity

Measuring how close two vectors are

Cosine similarity measures the angle between two vectors in the embedding space. It ignores magnitude and focuses purely on direction — which makes it ideal for comparing semantic meaning regardless of sentence length.

cosine_similarity(A, B) = A · B / (‖A‖ × ‖B‖)

Score interpretation

0.9 – 1.0Near-identical meaning
0.7 – 0.9Very similar, same topic
0.5 – 0.7Related but distinct
0.0 – 0.5Different topics
< 0Opposite or unrelated

Why it matters in RAG

When a user asks "Why did the payment service crash?", the retriever embeds that question and finds chunks with the highest cosine similarity from your knowledge base. Threshold is typically 0.7–0.8 for reliable retrieval.

Real example: "NullPointerException in OrderService" scores ~0.87 similarity with "null reference error in order module" — different words, same bug. Keyword search would miss this; embedding search finds it.
Reference

Vector dimensions by model

The dimension count determines the "resolution" of the semantic space. More dimensions can encode finer distinctions — but also mean more memory, slower similarity search, and higher storage cost. Embedding models are optimised for smaller dimensions; generative model hidden states run much larger.

Model Type Dimensions Relative scale Use case

Small (384–768)

Fast, memory-efficient. Ideal for RAG on commodity hardware. sentence-transformers all-MiniLM-L6-v2 (384 dim) can embed thousands of chunks per second on CPU.

Medium (1 024–3 072)

The sweet spot for production RAG. text-embedding-3-large (3072) is the current OpenAI recommendation. Can be truncated to 256 or 1024 dims with minimal quality loss.

Large (4 096–16 384)

Generative model hidden states. Not usually used as sentence embeddings directly — these are per-token representations inside the model during inference.

Storage math: 1M document chunks × 1536 dims × 4 bytes (float32) = ~6 GB of vector data. At 768 dims it halves to ~3 GB. Choose dimensions based on your retrieval quality requirement vs infrastructure cost.
Mental Models

Building the right intuition

GPS coordinates for meaning

A GPS coordinate (lat, lng) is a 2-number vector that places you on Earth. An embedding is a 768-number vector that places a sentence in "meaning space". Close coordinates = nearby places. Close vectors = similar meaning.

Dimensions = independent features

Each dimension loosely encodes an independent aspect of meaning — formality, topic domain, sentiment, technical specificity, etc. No single dimension is interpretable alone, but together they place the sentence precisely.

Training creates the geometry

The model isn't programmed with rules about Java errors. It learned from billions of examples that sentences appearing in similar contexts should have similar vectors. The geometry emerges from co-occurrence patterns.

Vector arithmetic works

The famous king − man + woman ≈ queen holds because relationships are encoded as vector offsets. In code embeddings: interface − concrete + abstract ≈ abstract class is a real measurable pattern.

What this means for your team: When building RAG on your Confluence + Jira knowledge base, the embedding model is doing the semantic heavy lifting. Your job is to choose the right chunk size, the right overlap, and the right similarity threshold — the vector geometry handles the meaning matching.
